# Agent Loop 操作规范

本文件定义 agent 在 SleepShip 循环中的具体行为规范——怎么工作、怎么协作、怎么在卡住时自救。

---

## 1. Agent Team 模式

不要单打独斗。用 Agent Team 并行推进：

```
主 Agent（Orchestrator）
├── Agent A: 后端实现（API、数据库、业务逻辑）
├── Agent B: 前端实现（页面、组件、样式）
├── Agent C: 测试 & QA（写测试、跑测试、端到端验证）
└── Agent D: 文档 & 配置（README、.env、部署脚本）
```

### 怎么用

- 用 `Agent` 工具分发子任务，每个 agent 独立工作
- 子任务之间无依赖的，**必须并行**启动（一条消息里发多个 Agent 调用）
- 有依赖的任务按顺序串行
- 每个子 agent 完成后，主 agent 汇总结果，决定下一步

### 任务拆分原则

- 每个子任务应该是独立可验证的（有明确的完成标准）
- 子任务粒度：1 个功能模块 或 1 组相关文件
- 不要把整个项目丢给一个子 agent

## 2. Skills 使用

在开发过程中主动使用以下 skills：

### PUA Debugging（必须使用）

当遇到以下情况时，**强制**触发 PUA debugging skill：

- 同一个错误修了 2 次还没修好
- 测试连续失败，你在反复调整同一段代码
- 你想说"我做不到"或"这个方法行不通"

PUA debugging 会逼你穷尽所有方案，不允许放弃。这是 SleepShip 的核心纪律——**agent 不许停，除非分数达标。**

### 其他推荐 Skills

| Skill | 什么时候用 |
|-------|-----------|
| `/investigate` | 遇到 bug，先调查根因再修 |
| `/simplify` | 代码写完后审查是否有冗余 |
| `/review` | 提交前自审代码质量 |
| `/qa` | 功能完成后端到端测试 |
| `/browse` | 需要测试 Web 页面的交互 |

## 3. 插件 & 工具

充分利用可用的 MCP 插件和工具：

### 浏览器测试
- 用 Playwright / browse 工具做 UI 测试
- 截图对比验证页面是否正确
- 测试表单提交、路由跳转、响应式布局

### Telegram 通知
- 关键节点主动通过 Telegram 汇报进度
- 遇到需要人工决策的问题，发 Telegram 问用户
- 不要在等用户回复时空转——先做能做的事

### 代码搜索
- 用 Grep / Glob 搜索代码，不要靠猜
- 修改前先读懂上下文，不要盲改

## 4. 失败处理纪律

### 永不放弃原则

SleepShip agent 的核心原则是：**在分数达标之前，不允许停止。**

遇到困难时的处理顺序：

```
第 1 步：读错误信息，理解根因
        ↓
第 2 步：搜索代码库，找到相关上下文
        ↓
第 3 步：尝试修复
        ↓
第 4 步：修复失败？换一种方案
        ↓
第 5 步：3 种方案都失败？触发 PUA debugging
        ↓
第 6 步：PUA 也搞不定？拆分问题，分多个子 agent 各攻一面
        ↓
第 7 步：仍然无解？通过 Telegram 向用户求助，同时继续做其他任务
```

### 禁止行为

- **禁止**说"这个我做不了"然后停下
- **禁止**在同一个方案上重试超过 3 次（换方案）
- **禁止**跳过测试说"应该能用"
- **禁止**不跑端到端测试就声称功能完成

## 5. 每轮工作流程

每一轮（turn）agent 应该：

```
1. 读 GOAL.md，确认当前还差什么
2. 读最新的 review 反馈（.sleepship/reviews.jsonl 最后一条）
3. 确定本轮优先级：
   - 有 bug？先修 bug
   - 没 bug？做最低分维度
   - 都不低？补功能清单里未完成的
4. 开工（优先用 Agent Team 并行）
5. 完成后自测：
   - 跑 build
   - 跑 test
   - curl 关键 endpoint
   - 浏览器测试（如果是 Web 项目）
6. Git commit（有意义的 commit message）
7. 等待 reviewer 评分
```

---

## 6. 端到端测试

每轮结束前，必须做端到端验证。不要只跑单元测试。

### 验证清单

- [ ] 项目能一条命令启动，无报错
- [ ] 核心用户流程走通（主流程端到端跑一遍）
- [ ] 所有功能是真实实现（无 mock、无 fallback、无 TODO）
- [ ] 错误情况有合理处理（不会崩溃或静默失败）
- [ ] 模块间联动正确（A 的输出作为 B 的输入能正常工作）

### 使用资产

端到端测试时使用 `.sleepship/.asset/` 里的资产：

```bash
# 加载环境变量
source .sleepship/.asset/.env

# 链上项目：用测试钱包做交易测试
# SaaS 项目：用真实 API key 测试第三方集成
# Bot 项目：用测试 token 发真实消息
```

资产是用户提前提供的真实凭证，用它们做真实的端到端测试，不要 mock。
